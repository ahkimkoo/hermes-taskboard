# Hermes Task Board v0.1.0

**First public release** — 2026-04-19

A Trello-style kanban that drives [Hermes Agent](https://github.com/NousResearch/hermes-agent) in batches. Single Go binary, SQLite + filesystem storage, Vue 3 frontend (no build step), bilingual UI (English / 简体中文).

## What's in this release

### Board & task workflow
- Four-column kanban: Draft → Plan → Execute → Verify → Archive, with drag-to-reorder within a column and drag-to-transition across columns.
- Priority badges (P0–P3) and colored tag chips on every card.
- Task dependencies: a task in Plan can list other tasks it depends on; the scheduler waits until dependencies reach their required state before moving it into Execute.
- Sequential or parallel dispatch: the scheduler watches server-level and profile-level concurrency caps and only fires what fits.

### Hermes integration
- Pluggable `hermes_servers` list — register any number of Hermes gateway instances, each with its own API key (encrypted at rest), models, and concurrency caps.
- Per-profile concurrency limits on top of per-server limits.
- Every task dispatch creates an **Attempt**: a live conversation with Hermes. Stream output is persisted as NDJSON and broadcast over SSE to the browser.
- In-card chat: send follow-up messages to continue an Attempt (**Ctrl/⌘ + Enter** to send).
- Two-click **Stop** on running Attempts to avoid accidental cancels.

### Scheduled tasks
- Friendly schedule picker with preset modes:
  - Every N minutes / every N hours (N is a free-form number)
  - Daily at HH:MM
  - Weekly on picked days at HH:MM
  - Monthly on a chosen day at HH:MM
  - Advanced (raw 5-field cron) for power users
- The picker shows a live preview of the cron expression it will save, and displays already-saved schedules in human-friendly prose ("每 15 分钟", "Weekly Mon, Wed, Fri at 09:00").
- Backend stores only standard 5-field cron (robfig/cron/v3). Any legacy `interval` rows in a pre-release DB are rewritten to cron on first boot; the worker also rehydrates missing `next_run_at` timestamps at startup.

### Tag system prompts
- Every tag can carry an optional **System Prompt**. When a task is dispatched, the prompts of all tags on that task are concatenated onto the base persona passed to Hermes — a task tagged `notify-qq` with prompt "post a short summary to QQ when done" inherits that behaviour automatically.

### Reliability
- **Orphan attempt reaper**: on startup, any Attempt still marked `queued` or `running` (left over from a crash or forced restart) is flipped to `failed` with a system event, so the UI doesn't spin forever on a dead Attempt and concurrency slots don't leak.
- **Nightly filesystem reaper**: `data/attempt/*` directories older than 90 days with no matching DB row are swept away.

### UX polish
- Bilingual UI that hot-swaps on the Preferences dropdown (English / 简体中文).
- Dark / light theme toggle.
- Electric chase borders call attention to active cards:
  - green+red — attempt is actively running
  - orange+red — attempt needs user input (or the task is sitting in Verify)
- Chat-style event stream: auto-scrolls while you're pinned to the bottom; if you scroll up to read history the pane stops dragging you down and surfaces a "↓ new messages" pill.
- Drag-vs-click on cards is now unambiguous — dragging no longer also opens the card modal.
- Image uploads are gated behind an Aliyun OSS config (the local server can't host images Hermes would actually be able to reach).

## Quick start

```bash
tar -xzf hermes-taskboard-v0.1.0-linux-amd64.tar.gz
cd hermes-taskboard-v0.1.0-linux-amd64
# First boot writes data/config.yaml with defaults; copy from config.example.yaml
# to customise (Hermes base_url, api_key, models, concurrency caps).
./hermes-taskboard -data ./data
# → browse to http://127.0.0.1:1900
```

A Docker image is also published; see `Dockerfile` in the repo.

## Upgrading from pre-release main

If you were running a build from `main` before this tag, simply drop in the new binary and restart — all schema migrations are idempotent. On first boot the new binary will:

1. Rewrite legacy `kind='interval'` rows in `task_schedules` to equivalent cron expressions (e.g. `15m` → `*/15 * * * *`) and clear their `next_run_at`.
2. Recompute `next_run_at` for any enabled schedule with a NULL timestamp.
3. Mark any `queued` / `running` Attempt left behind by the old process as `failed` (you'll see a startup log line `reaped orphan attempts count=N`).

No manual migration is required.

## Known limitations

- Hermes connections are not resumed after a taskboard restart — Attempts mid-stream at shutdown are marked failed. The conversation history is preserved on disk; send a new message to continue.
- Only one authenticated user per deployment (or auth disabled). Multi-user ACLs are not part of this release.

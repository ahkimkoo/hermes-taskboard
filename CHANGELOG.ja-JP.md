## 2026-04-19

### リリース（ラウンド 7.2 → v0.1.0 タグ付け）— スケジュールピッカー UX、孤立 Attempt の回収、ドラッグ/クリック修正

**スケジュールピッカーが自然言語に対応し、バックエンドは cron のみとなりました。**
以前はタスクごとのスケジュールピッカーが2つの種類（`interval` と `cron`）を公開し、`15m` や `0 9 * * 1-5` などの生の仕様を入力する必要がありました。cron は既に間隔を表現できる（`*/15 * * * *`）ため、2つ目の種類は冗長であると同時に使いにくいものでした。ピッカーをプリセット駆動に再設計しました：「N 分/時間ごと」（N は自由入力可能な数値）、「毎日 HH:MM」、「選択した曜日に毎週 HH:MM」、「毎月 D 日 HH:MM」、そして生の cron 用の Advanced エスケープハッチ。保存されたスケジュールは人間の読みやすい文言（「15 分ごと」、「Weekly Mon, Wed, Fri at 09:00」）で表示され、その下に検査用の生の cron が表示されます。ピッカーは保存される cron のライブプレビューを表示するため、ユーザーは何がディスクに書き込まれるかを正確に把握できます。

バックエンドは `kind='cron'` のみを受け入れるように書き直されました（API は他のものを拒否します）。起動時の1回限りの DB マイグレーションにより、レガシーな `interval` 行（`time.ParseDuration` 文字列）をベストエフォートで cron に近似変換します：`N` 分（最大 59）→ `*/N * * * *`；1 時間単位の倍数（最大 23）→ `0 */H * * *`；1 日を超えるものは毎日午夜にフォールバックします。マイグレーションされた行は `next_run_at` がクリアされ、ワーカーは新しい `ListEnabledNullNextSchedules` スイープにより起動時にそれらを再取得するため、アップグレード後にスケジュールがサイレントで見逃されることはありません。

**起動時の孤立 Attempt 回収（Orphan Attempt reaper）**
このリリースより前は、タスクボードプロセスが Attempt の処理中にクラッシュまたは強制終了された場合、Attempt の DB 行が永久に `running` のままでした — プロセスが所有しておらず、状態を切り替えるコードもありませんでした。UI はスピンのまま、同時実行スロットはリークし、何も回収されませんでした。修正：`main.go` からスケジューラ/cron ワーカーの起動*前*に新しい `Runner.ReapOrphans()` を呼び出します。`state IN ('queued','running')` の Attempt をスイープし、システムイベント `error: "process restart — attempt reaped as failed (no active runner)"` を `prior_state` と共に書き込み、状態を `failed` に切り替え、SSE 経由で状態変更をブロードキャストし、`Board.MaybeAdvanceAfterAttempt` を呼び出します。`needs_input` の Attempt はそのまま残されます — これらは正当にユーザー入力を待っており、入力を受信すると `SendMessage` が既にループを再起動します。

**ドラッグしてもカードモーダルが開かなくなりました**
カードのクリックハンドラーは `this.$el.style.display === 'none'` を「ドラッグ進行中」のガードとして使用していましたが、`drag.js` の `end()` は ブラウザが合成クリックを発火する*前*に `display = ''` を復元するため、このガードはドラッグ終了時に発火されるクリックを常に捉え逃していました — カードを移動するたびにタスクモーダルが開いていました。堅牢な `_dragStarted` フラグに置き換えました：各 `pointerdown` の開始時に `false` にリセットされ、`drag.start()` が呼び出された瞬間（移動距離 > 5 px）に `true` に設定され、`onClick` で設定されていれば消費されます。DOM のタイミングに関係なく動作します。

**v0.1.0 タグが最初のパブリックリリースを切ります。** 完全な出荷概要については `docs/release-notes/v0.1.0.md` を参照してください。

### ホットフィックス（ラウンド 7.1）— ユーザー自身のメッセージがイベントストリームで非表示になっていた

本日早些の SSE 修正（すべてのペイロードに `event` キーをラップして addEventListener/onmessage の不一致を回避する）が、AttemptRunner が各 NDJSON 行に設定していた内側の `event` フィールドを上書きしていました — そのため、`user_message`、`run_start`、`run_end` などがすべて `event: "event"` としてワイヤ上に到着し、UI が区別できませんでした。結果：フォローアップメッセージを入力すると、アシスタントの返信は表示されるものの、ユーザー自身のバブルがサイレントにドロップされていました。
修正：ペイロードがすでにイベント名を持っていない場合にのみ外側のラッパー名をマージし、ランナーの内側イベント名をそのまま保持します。この特定の障害モードが再発しないよう `test_sse_preserves_event_name` を追加しました。スイートは 30/30 グリーンです。

### 主要ラウンド 7 — SSE 修正、UX ポリッシュ、タグプロンプト、スケジュールタスク

**自動更新バグ（サイレントだった）**
古い SSE 配線バグを発見して修正しました：バックエンドの `writeSSE` は常に `event: <name>` フレームを出力していましたが、フロントエンドの `EventSource` は `onmessage` のみをリスンしていました（名前付きイベントでは発火しない）。結果：すべてのボードレベルイベント — task.moved、attempt.created、attempt.state_changed、preferences_updated — がサイレントにドロップされ、ボードはユーザーがリロードしたときのみ更新されていました。修正：`event:` ヘッダーを削除し、イベント名を JSON ペイロードにマージして、すべてが `onmessage` 経由で到着するようにしました。修正後、Verify → Execute への遷移（Verify でユーザーがフォローアップを送信すると自動的にトリガーされる）がリアルタイムでカードを列間で移動し、他のすべての状態変更がリロードなしで伝播するようになりました。

**カードアニメーション**
Verify / needs-input アニメーションはもはやチェイスではありません — 遅いゴールド ↔ ウォームホワイトの*呼吸する*ボーダー（3.5 秒 ease-in-out）になり、「一時停止、待機中」という印象を与え、「緊急性のある実行中」ではなくなりました。Execute 列の Running カードはエレクトリックグリーン+レッドのチェイスを維持しているため、2つの状態は一目で明確に区別できます。

**イベントストリーム：チャットスタイル自動スクロール**
EventStream はユーザーが下部にピン留めされているかどうかを追跡するようになりました。ピン留め中は、新しい出力（ストリーミングトークン、ツール呼び出し）が自動的にスクロールして表示されます。ユーザーが履歴を読むために上にスクロールすると、ストリームは強制的に下へドラッグするのをやめ、ペインの下部に「↓ 新着メッセージ」ピルを表示します；それをクリックする（または下部までスクロールし直す）ことで自動スクロールが再有効化されます。

**チャット入力**
送信は通常の Enter の代わりに **Ctrl/⌘ + Enter** になり、入力の下に小さなヒント（「Ctrl/⌘ + Enter で送信」）が表示されます — 通常の Enter は誤って送信せずに改行するために使えるようになりました。**停止**は2回クリック確認制です：最初のクリックで「停止を確認しますか？」を準備し、2回目のクリックで実際にキャンセルします；無視された場合は4秒後に自動リセットされます。

**Attempts 見出しの `?` ヘルプポップオーバー**
「Attempts」の横にある小さな `?` ボタンをクリックすると、「Attempt = 1回の実行；タスクは再実行できますが通常1回で十分です；既存の Attempt を継続するにはメッセージを送信してください」という1段落の説明がポップアップします。

**タグシステムプロンプト**
新しい `tags.system_prompt` カラム（冪等マイグレーション）。設定 → **タグ** タブでユーザーはタグを直接管理でき、オプションの `System Prompt` テキストエリアが含まれます。タスクがディスパッチされると、すべてのタグのシステムプロンプトが Hermes の `/v1/responses` 呼び出しに渡される基本ペルソナに連結されます。要件からのユースケース：プロンプト「When this task finishes, post a short summary to QQ」を持つ `notify-qq` タグ — この方法でタグ付けされたタスクは自動的にその指示を継承します。複数のタグは順序通りにスタックされます。

**スケジュールタスク（cron + interval）**
新しい `task_schedules` テーブル + `internal/cron` ワーカー（別 goroutine、5 秒ごとにティック）：
- **Interval** 種類：標準の `time.ParseDuration` 仕様 — `15m`、`2h`、`1h30m` — 最低 10 秒。前回の発火から N 後に再度発火します。
- **Cron** 種類：標準の5フィールド `min hour dom month dow`（robfig/cron/v3）。
- タスクごとに任意の数のスケジュールを設定でき、それぞれが独立に有効/無効にできます。
- 発火時、通常の Runner を介して新しい Attempt を作成します（同時実行ゲート + タグプロンプトがすべて適用されます）。
- 作成：`POST /api/tasks/{id}/schedules`、有効/無効の切り替え：`PATCH /api/schedules/{id}`、削除：`DELETE /api/schedules/{id}`。
- タスクモーダル内にレンダリングされる新しい `SchedulePicker` コンポーネントが種類、仕様、次回発火、オン/オフ切り替えを表示します。

**テスト**
5 件の新しい Playwright ケース（`test_tag_system_prompt`、`test_schedule_roundtrip`、`test_schedule_picker_ui`、`test_input_hints`、`test_attempt_help_popover`）。スイートは **29/29** グリーンです。

### ポリッシュ（ラウンド 6）— アニメーションカードボーダー
- **実行中のタスク（Execute 列）** に**エレクトリックグリーン+レッドの「チェイス」ボーダー**を追加：2つの狭い緑→赤グラデーションの弧がカードの周囲を 3 秒/周 で回転し、間に透明なギャップがあるためチェイスが明確に読み取れます。
- **Verify / needs-input タスク**に**同じチェイスですが、オレンジ+レッドで** — カードがあなたの注意を必要としていることを、アラートほど大声で叫ばずに知らせます。
- `conic-gradient(from var(--glow-angle), transparent, color, color, transparent, …)` を使用して実装し、2px の透明ボーダーに `background-clip: padding-box, border-box` で描画されます。`--glow-angle` は `@property` で登録されているため、ブラウザは 60 fps で滑らかに補間できます。
- バックエンド：`tasks` API が `active_attempts` と並行して `needs_input_attempts` を公開するようになり、フロントエンドは「実行中だが正常」と「実行中が入力待ちでブロック」を区別できます。
- `prefers-reduced-motion: reduce` の場合、静的な色付きボーダーにフォールバックします。
- Verify カードが `.needs-input` を受け取り、アクティブな Attempt を持つ Execute カードが `.running` を受け取り、どちらのクラスも同時に持たないことをアサートする新しいリグレッションテスト；計算された `animationName` / `animationDuration` が非ゼロであることを確認。スイートは **24/24** グリーンです。

### 修正（ラウンド 5）
- **Attempt リストの切り替えが実際に機能するようになりました**。以前のロジックは `attemptListVisible = listOpen || attempts.length > 1` だったため、タスクが 2 つ以上の Attempt を持つと、何度 Hide をクリックしてもリストが永久に表示されたままでした。現在、単一の `listOpen` フラグが可視性を制御します；`attempts.length > 1` の場合はデフォルトで開き、そうでない場合は閉じます。トグルボタンは Attempt が 1 つ以上ある場合常に表示されます。
- 設定 → 設定の各イベント切り替えの横に**サウンドプレビューボタン**を追加。▶ ボタンは、その特定のイベントが有効かどうかに関係なく対応するトーンを再生します（そのため、有効にする前に音を audition できます）。現在のボリュームドラフトを使用します。
- 2 件の新しい Playwright ケース；スイートは 23/23 グリーンです。

### UX 追加（ラウンド 4）
- **画像アップロードに画像ホスティングが必要になりました**：Hermes の `gateway/platforms/api_server.py` を読むことで、サーバーが `input` テキストをそのまま上流 LLM（DashScope など）に転送し、`image_url` コンテンツパーツをサイレントにドロップすることを確認しました。LLM は `http://127.0.0.1:1900/uploads/*` に到達できないため、現実的なセットアップではローカルストレージは役に立ちません。画像挿入ボタンと貼り付け/ドロップハンドラーは、Aliyun OSS が設定されていない限り非表示になりました；同じケースで `POST /api/uploads` は `503 image_upload_disabled` を返します。説明エディタの下にある役立つヒントがその理由を説明し、設定 → 連携を指し示します。
- **タスクモーダルがオーバーレイクリックで閉じなくなりました**：タスクオープンと新規タスクモーダルは、ヘッダーの明示的な ×（または新規タスクのキャンセル）でのみ閉じるようになりました。長い説明を編集中にモーダル周りの暗い領域を誤ってクリックしても、全体が破棄されることはありません。設定と確認モーダルは既存のオーバーレイクリック動作を維持します（ユーザー入力を失うリスクがないため）。
- テスト：3 件の新しいケース（`test_uploads_gated`、`test_task_modal_overlay_noclose`、`test_new_task_overlay_noclose`）；既存の `test_editor_controls` は OSS なしで Insert-image ボタンが存在しないことをアサートするようになりました。スイートは 21/21 グリーンです。

### UX 追加（ラウンド 3）
- **タグ入力**：タグは `tags` テーブル-backed のオートコンプリート付きチップベースコントロールになりました。これまでどのタスクで使用されたことのあるタグでも、将来のタスクの候補になります。Enter、`,`、Tab でコミット；チップの × または空の入力でのバックスペースで削除。（新しい `TagInput` コンポーネント。）
- **依存関係**：新規タスク/編集タスクフォームに **Depends on** ピッカーが追加されました。各依存関係は `{task_id, required_state}` で、状態は **Verify**（対象の Attempt が完了したら開始、ユーザーがまだ承認していなくても）または **Done**（人間の承認を待つ）のいずれかです。スケジューラの `AllDependenciesDone` はエッジごとに状態を尊重します。（新しい `DependencyPicker` コンポーネント、`task_deps.required_state` のスキーママイグレーション。）
- **必須 vs オプション** — フォームは **Title** の後に赤い ★ を、他のすべてのラベルの後にインラインの `(optional)` マーカーを表示するようになりました。Title が唯一の必須フィールドです。
- API：`/api/tasks` の作成/パッチは依存関係を `["id", ...]`（レガシー、暗黙の `done` ゲート）または `[{task_id, required_state}, ...]` のいずれかで受け入れます；バックエンドは正規化してより豊富な形で保存します。
- テスト：3 件の新しい Playwright ケース — タグ入力のオートコンプリート + チップ削除；`required_state=verify` の依存関係ピッカーのラウンドトリップ；新規タスクフォームのオプショナルマーカーの監査。スイートは 18/18 グリーンです。

### UX 全面刷新（ラウンド 2）
- **ドラッグ＆ドロップ**を pointer events 上に再構築：ソースカードは点線のプレースホルダーで隠され、スタイル付きの浮動クローンがカーソルに従い、ドロップは正確なスロット（「列」だけでなく隣接要素の間）にコミットされます。Trello により近く、ブラウザ HTML5 ドラッグゴーストとははるかにかけ離れた感触です。（要件 #1）
- **タスクの順序がユーザー制御可能になりました**：`tasks` テーブルに `position` カラムを追加し、既存 DB のための自動マイグレーション（`created_at` から position をシード）を伴います。新しいタスクは Draft 列の末尾に配置されます；ドラッグによる並べ替えは永続化され、リロード後も保持されます。リスト API は優先度でソートしなくなり、`(status, position)` で行を返し、クライアントは単純に反復します。（要件 #7、#8）
- **リッチ説明エディタ**：タイトルは必須、本文はオプションの Markdown で Write/Preview 切り替え付き。貼り付け、ドロップ、または画像の選択 — `POST /api/uploads` でアップロードされ、Markdown 画像参照がキャレット位置に挿入されます。（要件 #2）
- **画像ホスティング**：新しい `internal/uploads` パッケージ。デフォルトではローカルディスク（`data/uploads/`、`/uploads/{name}` で提供）、または設定されていれば Aliyun OSS。OSS 認証情報（`oss.access_key_id` + `access_key_secret`）は `data/config.yaml` に格納され、シークレットは Hermes API キーと共に AES-GCM で暗号化されて保存されます。設定ページに「Integrations」タブが追加されます。
- **Attempt リスト**：Attempt ごとにローカルフォーマットの開始時間 + 短い ID を表示します；Attempt が 1 つ（または 0 個）のみの場合は単一ペインレイアウトに折りたたみます；「+ New Attempt」ボタンは縮小され、新しい Attempt が別の同時実行スロットを消費することを説明する確認ダイアログの背後に配置されます。（要件 #3）
- **イベントストリームがセマンティックになりました** — 各 Hermes イベントはユーザーメッセージ、アシスタントバブル（Markdown レンダリング付き）、または名前/引数/出力を表示する折りたたみ可能なツール呼び出しカードにグループ化されます。生の JSON ダンプはなくなりました。（要件 #4）
- **ライト/ダークテーマ切り替え**をトップバーに追加（☾/☀）、`data/config.yaml` の `preferences.theme` に永続化されます。フルライトテーマパレットが CSS 変数で定義されています。（要件 #5）
- **削除ゲート**：*Delete task* ボタンはカードが Archive 列にある場合のみ表示されます；1 回クリックすると 2 つ目の「Confirm delete?」ボタンが表示されます。（要件 #6）
- **列のサブタイトル**：6 つの列それぞれに、その意味を説明する小さな灰色の1行が追加されました（例：Plan → 「Queued and ready for execution.」/ 計画 → 「排队准备执行」）。（要件 #10）
- **設定ページ**：*Models* の下に、各行が Hermes エージェントプロファイル（Hermes API が「model」と呼ぶものと同じ）に対応することを説明する明示的なヘルプ段落が追加されました。（要件 #11）
- **設定モーダル再オープンバグ** — 修正済み。常に `showSettings = false → true` の遷移を経由するようになり、2回目のクリックが no-op になる状態のウィンドウを回避します。（要件 #12）
- **i18n がリアクティブに書き直されました**（Vue ref）— 切り替え後の言語混合はもうありません。`$t(key)` ルックアップは一貫して1つの辞書に対して解決されます。欠落しているキーは英語にフォールバックし、残りの中国語文字列には決してフォールバックしません。（要件 #9）

### 舞台裏
- すべての `config.Preferences`/`Sound`/`Scheduler`/`Archive`/`Server`/`OSS` 構造体フィールドに JSON タグを付与し、API が PascalCase キーではなく `{ language, theme, sound: {…} }` を返すようになりました。これはフロントエンドに対するサイレント API 破壊でしたが、ここでまとめて修正しました。
- `POST /api/tasks/{id}/transition` が `after_id` / `before_id` を受け入れ、特定のドロップスロットを要求できるようになりました；バックエンドは隣接要素の中間に新しい `position` を計算し、衝突からの回復のために必要に応じて列の番号を振り直します。
- フロントエンドの新しいモジュールレイアウト：`i18n.js`、`markdown.js`、`drag.js`、`description-editor.js`、`event-stream.js` がそれぞれ個別のファイルになり、`app.js` によってインポートされます。

### 追加
- Hermes Task Board の初期実装。
- Vue 3 フロントエンドを埋め込んだ単一 Go バイナリ（`go:embed`）— Web アプリ用の個別のビルドステップは不要。
- SQLite + ファイルシステムストレージ（`data/db/taskboard.db`、`data/task/*.json`、`data/attempt/{id}/events.ndjson`）。
- ホットリロード対応 YAML 設定（`POST /api/config/reload`）と、AES-GCM 暗号化された Hermes API キーの保存。
- 6 つの固定列を持つ Kanban（`draft → plan → execute → verify → done → archive`）、HTML5 ドラッグ＆ドロップ、優先度 P1–P5、タグ、依存関係。
- タスク状態マシン；バックエンド自動遷移は `plan → execute`、`execute → verify`、`verify → execute` のみ。
- スケジューラ：5 秒ごと（設定可能）に `plan + auto + deps-done` タスクをスキャンし、3 つの同時実行ゲート（グローバル / サーバー / (サーバー, モデル)）を尊重。
- `AttemptRunner`：Attempt と Hermes の会話との 1:1 マッピング；メッセージキューイング、Verify フォローアップ後の再エントリー、SSE ストリームの消費を処理。
- Hermes クライアント（`internal/hermes`）：`CreateResponse`、`StreamRunEvents`、`CancelRun`、`Health`、`Models`。設定リロード時にプールを再構築。
- タスク、Attempt、タグ、Hermes サーバー（CRUD + 接続テスト + モデルインポート）、設定、認証のための REST API。
- SSE チャネル：`/api/stream/board` および `/api/stream/attempt/{id}`。ディスク上の NDJSON からの `Last-Event-ID` によるレジューム対応。
- オプションのユーザー名/パスワードログイン（bcrypt、HMAC 署名付き Cookie）。デフォルトはオフ；設定 → アクセスコントロールで有効化。
- i18n：英語 + 簡体字中国語、ライブで切り替え可能；文字列は `/locales/*.json` からロード。
- PWA：マニフェスト + サービスワーカー（アプリシェルキャッシュ）；API/SSE はネットワークファースト。
- Web Audio によるサウンドキュー（`execute_start`、`needs_input`、`done`）— オーディオアセットの配信は不要。
- レスポンシブレイアウト：6 列（≥1200 px）、3 列スクロール（768–1199 px）、トップタブ付き単列（<768 px）。
- `build.sh`、`release.sh`（クロスプラットフォーム tarball + チェックサム）、および distroless `Dockerfile`。
- 同じホストで実行されているライブ Hermes インスタンスに対してキャプチャされたスクリーンショット（`docs/screenshots/`）。
- 二言語（英語 / 中文）の README とこの CHANGELOG。

### 既知の制限
- マルチユーザーまたは RBAC なし（v1 ではシングルユーザーが設計上の意図）。
- ツール呼び出しの `approval_required` イベントは表示されますが、UI で対話的に承認することはできません — 現時点ではシステムイベントとしてレンダリングされます。
- スケジューラのサーバーヘルスショートサーキットは 30 秒のキャッシュを使用します；Hermes サーバーがティックの途中でダウンした場合、キャッシュが期限切れになるまで一時的に失敗した Attempt のストリームが表示される可能性があります。
- `archive.auto_purge_days` はスケジューラ設定で読み込まれますが、回収 goroutine はスタブのまま — ファイルは現在、タスクを手動で削除するまで蓄積され続けます。
